# Blackboard vs. Event-Driven Design for IOT Wearables
This repository serves as a means to compare two different architectural styles, namely a "blackboard" design and an "event-driven" design, and their practicality in real-world deployments on Internet of Things (IOT) wearable devices. For our simulation environment, we have created a synthetic smart watch, whose backend has been implemented twice, once per architectural style. This watch is responsible for capturing a wide variety of information about the environment and user's health via 9 sensors, which are controlled in different ways depending on the underlying architecture's capabilities, and then displaying that information to the user.

## Note About the Water Sensor
In order to effectively test our system's ability to detect and remove simulated water, while maintainging our human-readible JSON raw data format, the JSON data will continue to show the device as being water logged even after it has been cleared. In other words, for the sake of our simulation, where we cannot know the arbitrary time T in which the user will dispell the water, the JSON file of raw sensor data will continue to send a waterlogged signal to the IOT wearable; the wearable itself will simply ignore this sensor's waterlogged signal if water has been cleared at least one time during the simulation. In the real world, where the data is not fixed to a JSON file and events are truly random (i.e., they do not need to be deterministic for the sake of our clean/clear demonstration), this manipulation of the waterlogged flag would be removed, and instead the system would simply purge/detect water as expected.

## Brief Overview of Styles
While we discuss more in-depth the differences between our styles below, this section provides a quick-start to understanding what can be found in the code/implementation of each architecture.

**Blackboard**
- Central data store
- Constant polling of sensors
- Longer-term "memory" in comparison to event-driven
- Higher simulated power consumption in comparison to event-driven

**Event-Driven**
- No centralized data store
- Sensors are polled only when user apps explicitly requires it
- Shorter-term "memory" in comparison to blackboard
- Lower simuated power consumption in comparison to blackboard

## Detailed Explanation of Differences in the Architectural Styles Selected
### Blackboard Implementation
Our blackboard architecture is implemented as a centralized and mutable data repository. The ```Blackboard``` struct stores every sensor reading in per-sensor vectors (in the format ```heart_rate: Vec<f64>```, ```blood_pressure: Vec<String>``` etc.). This core pattern of interaction allows for a continuous accumulation of data each second -- as simulated data is "read" by the sensors from the JSON file (then place into a ```SensorFrame```), each incoming ```SensorFrame``` is applied via ```apply_frame()```, pushing each of the fields into its own respetive vector within the blackboard. At the same time, the blackboard also updates the boolean flag for water intrusion and deducts a fixed power cost per sensor polled. By designing the system in this way, it is able to make historial data into a first-class citizen, where the blackboard can retain the entire slate of data from the current runtime (as well as access it via the ```get_all_*() -> &Vec<f64>``` functions). As a result, all components -- such as a downstream analystics module or the UI -- can easily and on-demand retrieve not only the most recently recorded value but also the complete trend.

Similarly, the connectors between the blackboard and the sensors is direct and implicit. The controlling class pops frames from a queue and synchronously calls ```apply_frame()```, such that there is not a truly separate abstraction of the connector(s) (i.e., there is not event bus or message router). By coupling these features tightly together, the blackboard is able to demonstrate its unique features and benefits -- the blackboard itself becomes the sole intermediary, with all aspects of communication flowing to and from it via simply, ordinary function calls.

As a result, the blackboard design feels quick and snappy for a user to interact with. There is no spin-up times for activities like retrieving the user's heart rate, as that sensor data has already been automatically collected and incorporated into the data storage of the blackboard. While having a dedicated event-bus system might save on power and lead to increased data privacy (as discussed below in the Event-Driven Implementation section), the risk to the user of missing a transient event is likely to far outweight the cost of a bigger battery in the watch.

### Event-Driven Implementation
However, in contrast to the blackboard implementation, the event-driven design utilizes an Event Bus that is the centralized broadcast component that publishes events or messages out to its subscribers to separate the apps from the sensors. Instead of a mutable repository that all components read or write to, interactions are represented as events that get published or received by components through this Event Bus asynchronously.

The `Event Bus` struct uses a `tokio::sync::broadcast` Sender which allows multiple producers to send messages to multiple consumers, making it thread safe. Then Sender includes the `Event` enum struct which holds the event types such as `RequestSensor(String)`, `StopSensor(String)`, `SensorData(Sensors)`, `RequestHealth`, `StopHealth`, etc. The `EventBus` struct is cloneable with the `#[derive(Clone)]` which means the EventBus can be cloned and a new sender handler can be created that still points to the same broadcasting channel of the event bus so that multiple components can share access without conflicts. Basically allowing anyone who has a clone of the bus to be able to be a subscriber/receiver to the bus. In the constructor (`new()`) of the EventBus, the channel is defined with a capacity of 300 messages. This means a buffer of 300 messages/events can be queued, but if more than 300 events enter/exist in the channel then they will be overwritten and lost. For this simulation, 300 seemed like a safe number to ensure no overload of events occurred. Then the `publish()` function is called by sensors and apps to publish an event and the `subscribe()` function is called by apps and other sensor management services to receive streams of events.

Since subscribers to the bus receive all events, they filter through the incoming stream to decide whether or not action needs to occur. Apps are one of the subscribers of the bus which can be seen within each `run_*app name*_app()` function, there is a `bus.subscribe()`; Each app was separated into their own modules to support the different sensors and items that needed to be displayed into the UI. This also made it simpler for the frontend to connect to the backend because all the logic of the sensors are stored within each backend app module that uses them. The apps are the weather_app.rs, health_app.rs, message_app.rs and water_removal_app.rs.

The Weather App uses an `async run_weather_app(bus: EventBus)` function to subscribe to the bus and the loop continuously filters incoming events to see if they match the `RequestWeather`, `StopWeather` or `SensorData(String)` events that the weather app uses. The `RequestWeather` event is when a user opens the weather app, so it sets that sensor to active then publishes the request for the “Temperature Outside” and “Barometric Pressure” sensors. All the apps use the match event in the loop to allow for easy addition of more events. They all also use the `recv().await`, which is an async operation that ensures no data is returned if no messages are in the channel. The `StopWeather` request is when a user closes the app so the sensor is `active=false` and the `StopSensor` for “Temperature Outside” and “Barometric Pressure” is requested to be idle. The `SensorData(sensor)` event is for when that requested data is received, it checks that the sensor is active (user is still in the app), that the temperature/barometric sensor data exists, then prints the received values.


Similarly, the Message App uses an `async run_message_app(bus: EventBus)` function to also subscribe and filter incoming events. There is a `messages: Vec<String>` that keep all messages that come in to display the whole conversation. When the app is opened, `RequestMessages` is the matched event that requests the “Bluetooth Transmissions” sensor. When closed, `StopMessages` is matched, the sensor is set to `active=false` and the “Bluetooth Transmissions” is requested to be idle. The `SensorData(sensor)` event is also for when that requested data is received, it checks that the sensor is active (user is still in the app), that the messages sensor data exists, then prints the received values.


Also following the same pattern, the Health App uses the `run_health_app(bus: Eventbus)` function. One difference from the other two apps is that it tracks the `total_steps` and `total_distance` instead of returning the single frame value that comes from the sensor. This is to simulate the total steps or distance the user would want to see over the course of an amount of time instead of decreasing by the second which is unrealistic. The sensors used in this app are “Heart Rate”, “Blood Pressure”, “Steps Taken”, “Distance Traveled” and “Time”. Everything else is the same as discussed in the other two apps.


Lastly, the Water Removal App has a couple of differences as well. It also runs the `async run_weather_removal_app(bus: EventBus)` but it sets a `water_present = false` and `water_cleared = false` flag. This flag is used when the `WaterDetected` event comes in, it notifies the user with a console message. Then there is a `ClearWater` event that sets `water_cleared = true`, just to simulate the water was cleared and this happens at least once in the simulation.

Sensor behavior is only activated/deactivated based on logic from the sensor manager module. The SensorManager has an `async run_sensor_manager(bus: EventBus, loader: Arc<Mutex<SensorLoader>>)` function that maintains a `HashSet<String>` of active sensors and only publishes data for those active sensors. It also tracks the power of the device with the `total_power = 1000` to start with. Every second a sensor is active, it removes 1 power unit per second of data that is taken from the source. This design shows an optimal power conservation since sensors are not active all the time like the blackboard. The sensor manager loops through the received requests for what sensors to activate or deactivate. It grabs the snapshots of requested data from the SensorLoader that is already converted to the `SensorData(sensor)` and publishes it to the bus.

The `SensorLoader` struct in the sensor_loader.rs is designed to load and iterate the JSON data from sensor_data.json using a `Vec<HashMap>` for quick lookup and ordered data. The implementation of SensorLoader reads the JSON and returns errors if it could not be read. Then it loads snapshots, maps it to the appropriate sensor and data type, then that gets returned to the Sensor Manager.
The Arc<Mutex<SensorLoader>> is used to ensure mutable, shared, async access to the same SensorLoader instance through multiple tasks safely. This is important we wouldn't want multiple tasks to read/modify a snapshot of data or it could cause conflicts.
Event and Sensors are the enums for this system. It was better to use enums instead of structs because enums are mutually exclusive. Sensors and Events will not have every value type defined or used in every user interaction so enums made more sense. For all Events to have to be communicated through the `Event::<EventType>`, this also decoupled the components from the publishers/subscribers of the events.

The Client struct has a `bus:EventBus` field, making it a small abstraction that exposes high level functions to publish events on behalf of the UI. This shows encapsulation because the UI doesn’t need to know event names and this ensures only valid types are published. Instead of the UI manually constructing events or directly reaching each sensor needed, it uses the Client to request high level events like `RequestHealth` for opening the Health App that will follow the rest of the process as already described above to get that relevant sensor data. The Client would be initialized and these functions used to open/close each app (and trigger the `clear_water` function) would be used as apis from the frontend to begin the flow of events.

Overall, the event-driven has no store or retention of past data and the concurrency makes the process a little more complicated. Everything happens through events which has a main channel or event broker to handle those events. The benefits result in better power efficiency, scalability for events/apps, and real-time processing of events for the user. This should allow the user to notice real-time data when they open the app. Some drawbacks include possibility of missed events from overflowing of channel capacity or general network issues. This would be hard to track and debug too. The possibility of unordered events is also a drawback. There has to be checks to make sure events are handled in order, but even so it's possible a Close App event could be received before an Open App (in this simulation), creating issues with state accuracy/consistency across the system.

## Detailed Explanation of Rationale Behind Final Architecture Selection
After careful review of the data gathered during development and experimentation with our system, we have determined the blackboard architecture to be the most effective at meeting our IOT wearable's real-world needs. 
The Blackboard architecture remains the most suitable design for the IoT smartwatch, as its fundamental characteristics align with the core requirements of health monitoring and UI responsiveness, whereas the Event-Driven architecture introduces risks that fundamentally compromise the system's reliability.

## 1. Event-Driven Risks Compromise Core System Goals
The primary justification for considering an Event-Driven architecture was to minimize idle power consumption and maximize battery life. However, the risk analysis reveals that the current implementation actively undermines this goal. Because the system utilizes a tokio::sync::broadcast channel, every published SensorData event awakens every application to filter the incoming data. As the system scales and more applications are added, this constant broadcast processing will result in severe CPU overhead, negating the intended power-saving benefits. 

Furthermore, the Event-Driven model introduces unacceptable risks to data integrity. The smartwatch's value relies heavily on capturing transient physiological events and maintaining reliable long-term time series data for health risk detection, such as arrhythmia. The tokio broadcast channel's fixed buffer limit of 300 messages creates a "Slow Consumer" vulnerability. If the data access layer receives sensor data faster than the applications can process their wake-and-filter cycles, historical events will be overwritten and permanently lost. Combined with the risk of out-of-order message delivery corrupting the active/asleep state of the sensors, the Event-Driven model cannot guarantee the precise, continuous data stream required by the Health application.

## 2. Blackboard Risks are Solvable Implementation Details
While the Blackboard architecture carries notable risks, they are primarily structural implementation flaws that can be mitigated within the chosen Rust technology stack, rather than inherent architectural dead-ends. Concurrency and Data Races: The lack of concurrency protection when multiple sensors call apply_frame() is a critical but easily solvable defect. Wrapping the Blackboard's internal state in a Mutex or RwLock will guarantee thread safety and prevent data corruption without requiring an architectural overhaul. Unbounded Memory Growth: The risk of vectors appending indefinitely and exhausting the wearable's limited memory can be resolved by transitioning the internal Vec structures into fixed-size circular buffers (e.g., VecDeque with a strict capacity). This ensures the device retains the necessary sliding window of historical data for UI smoothing and trend analysis without causing a fatal memory overflow.Coupling and Single Point of Failure: While a centralized data store represents a single point of failure, this is an acceptable tradeoff in an embedded wearable environment. If the central process crashes, the OS can simply restart the service. The tight coupling to a concrete in-memory store is a functional advantage for UI development, as it guarantees that applications like the watch face and Health dashboard have instant, zero-latency access to the data.

## Conclusion
The smartwatch's primary value to a user is its immediacy. Users expect health metrics, weather data, and notifications to be instantly available without warm-up latencies or missed events. The Blackboard architecture inherently supports this by maintaining a continuously updated, queryable state. By patching the Blackboard's memory and concurrency vulnerabilities, the system will deliver the robust, highly responsive user experience required by the front-end applications, avoiding the data loss and CPU-spiking pitfalls of the Event-Driven broadcast model.
